# Quality Gap Tracking (QBASE-002)

QBASE-002 defines the requirement to track ownership for every open quality gap.

## Requirements

Every quality gap entry must include:

1. **Owner** - GitHub username of the person responsible for remediation
1. **Target Date** - ISO 8601 date when the gap should be closed
1. **Status** - Current state (`Open`, `In Progress`, `Closed`)

Additional required fields:

- **Severity** - `Critical`, `High`, `Medium`, or `Low`
- **Baseline Metric** - Reference to the specific metric from `baseline.v1.json`
- **Description** - Clear explanation of the gap
- **Action Items** - Checklist of specific remediation steps

## Document Structure

Quality gaps are tracked in three documents:

### quality/gaps.md

The primary gap tracking document containing:

- **Open Gaps:** All active quality issues requiring remediation
- **Closed Gaps:** Historical record of resolved issues with resolution details

Each gap entry follows this template:

```markdown
### GAP-XXX: Short Title

**Severity:** Critical | High | Medium | Low
**Owner:** @username
**Target Date:** YYYY-MM-DD
**Status:** Open | In Progress | Closed
**Baseline Metric:** Package name and metric reference

**Description:**
Detailed explanation of the quality gap.

**Action Items:**
- [ ] Specific remediation task 1
- [ ] Specific remediation task 2

**Related Issues:** #issue-number (or TBD)
**Resolved By:** #PR-number (for closed gaps only)
```

### quality/backlog.md

Weekly-updated backlog for sprint planning:

- Current sprint focus
- Next sprint planning
- Prioritized backlog by severity and target date
- Completed work log

### quality/baseline.v1.json

The source of truth for quality metrics. Generated by `pnpm run quality:baseline`.

## Validation

Run `pnpm run quality:validate-gaps` to verify:

- All open gaps have required fields (owner, target_date, status)
- Target dates are in ISO 8601 format
- Severity is one of the allowed values
- All gaps reference baseline metrics
- Closed gaps include resolution PR references

## Workflow

### Identifying Gaps

1. Run `pnpm run quality:baseline` to generate current metrics
1. Compare against coverage targets (80% minimum)
1. Create gap entries for any metric below threshold
1. Assign owner and target date

### Closing Gaps

1. Complete remediation work
1. Verify metrics meet targets with `pnpm run quality:baseline`
1. Update gap status to `Closed`
1. Add `Resolved By: #PR-number` with link to the fix
1. Move entry from "Open Gaps" to "Closed Gaps" section
1. Update weekly backlog

### Weekly Review

Every Monday:

1. Review `quality/gaps.md` for upcoming target dates
1. Update `quality/backlog.md` with current sprint status
1. Adjust priorities based on severity and dependencies
1. Communicate blockers and progress

## Coverage Targets

All packages must meet minimum thresholds:

- **Lines:** 80%
- **Statements:** 80%
- **Functions:** 80%
- **Branches:** 80%

Gaps are automatically identified when any package metric falls below these thresholds.
