# No-New-Debt Rule (MIG-002)

**Requirement:** MIG-002 (Phase 1 of Quality Migration)

**Parent Spec:** Quality Baseline Migration Strategy

## Overview

The No-New-Debt Rule prevents new quality violations from entering the codebase after the baseline lock (MIG-001). This is Phase 1 of the quality enforcement migration, implementing diff-aware checks that allow existing technical debt while blocking new violations.

## Acceptance Criteria

1. ✅ CI diff-aware checks identify new violations not present in the baseline
1. ✅ Existing baseline violations outside changed scope remain visible but non-blocking
1. ✅ New violations in changed files/scope FAIL CI
1. ✅ Checks compare current state against locked baseline (`quality/baseline.v1.json`)

## Implementation

### Script

`scripts/check-no-new-debt.mjs` implements the no-new-debt rule with the following checks:

1. **Lint Diagnostics**
   - Compares current lint errors/warnings against baseline counts
   - New lint violations block CI
   - Existing baseline violations are reported but don't block

1. **Test Violations**
   - Tracks skipped tests (`.skip` markers)
   - Tracks flaky tests (intermittent failures)
   - New skipped or flaky tests block CI

1. **Coverage Regressions** (diff-aware)
   - Checks coverage per package: lines, statements, functions, branches
   - Only checks packages with changed files
   - Coverage drops > 0.1% from baseline block CI
   - Allows rounding tolerance of 0.1%

### Exit Codes

- `0`: No new violations detected → CI passes
- `1`: New violations found → CI fails

### Environment Variables

- `BASE_BRANCH`: Branch to compare against (default: `main`)

## Usage

### Local Development

```bash
# Check for new violations against baseline
pnpm run quality:check-no-new-debt

# Regenerate baseline (intentional quality changes)
pnpm run quality:baseline
git add quality/baseline.v1.json
git commit -m "chore(quality): update baseline after intentional changes"
```

### CI Integration

The check runs automatically in CI via the `quality/no-new-debt` job in `.github/workflows/ci.yml`.

#### Required Branch Protection

Add `quality/no-new-debt` to the required status checks in `.github/branch-protection.main.json`.

## Diff-Aware Behavior

The no-new-debt check is **diff-aware** for coverage regressions:

- Only packages with changed files are checked for coverage drops
- If `git diff` cannot determine changes (e.g., comparing same branch), all packages are checked
- Lint and test violations are always global (not diff-aware)

### Example

If you change `packages/core/src/index.ts`:

- ✅ Coverage regression in `core` package blocks CI
- ✅ Coverage regression in `cli` package is ignored (not changed)
- ✅ New lint errors anywhere in the codebase block CI

## Violation Types Checked

| Type | Diff-Aware | Baseline Source | Failure Condition |
|------|------------|----------------|-------------------|
| Lint Errors | No (global) | `lintDiagnosticsBySeverity.error` | Current > Baseline |
| Lint Warnings | No (global) | `lintDiagnosticsBySeverity.warning` | Current > Baseline |
| Skipped Tests | No (global) | `skippedTests` | Current > Baseline |
| Flaky Tests | No (global) | `flakyTests` | Current > Baseline |
| Coverage (lines) | Yes (per-package) | `coverageByPackage.{pkg}.lines.pct` | Current < Baseline - 0.1% |
| Coverage (statements) | Yes (per-package) | `coverageByPackage.{pkg}.statements.pct` | Current < Baseline - 0.1% |
| Coverage (functions) | Yes (per-package) | `coverageByPackage.{pkg}.functions.pct` | Current < Baseline - 0.1% |
| Coverage (branches) | Yes (per-package) | `coverageByPackage.{pkg}.branches.pct` | Current < Baseline - 0.1% |

## Relationship to Other Quality Controls

### MIG-001: Baseline Lock Before Enforcement

The no-new-debt rule depends on a locked baseline generated by MIG-001:

- **MIG-001** establishes the baseline snapshot
- **MIG-002** enforces no regressions against that baseline

### QBASE-002: Quality Gap Tracking

Existing baseline violations should be tracked in `quality/gaps.md`:

- Each violation should have an owner, target date, and status
- The gap tracking document provides remediation plans for baseline debt

### Phase 3: Hard-Gate Rollout

MIG-002 is a stepping stone to Phase 3 hard-gate enforcement:

- **Phase 1 (MIG-002)**: No new violations (allows existing debt)
- **Phase 2**: Gradual remediation of baseline violations
- **Phase 3**: Hard-gate enforcement (zero violations allowed)

## Intentional Baseline Updates

If you intentionally change quality metrics (e.g., adding legitimate skip markers for flaky tests):

1. Regenerate the baseline:

   ```bash
   pnpm run quality:baseline
   ```

1. Review the diff in `quality/baseline.v1.json`

1. Commit with justification:

   ```bash
   git add quality/baseline.v1.json
   git commit -m "chore(quality): update baseline - added skip for flaky test #123"
   ```

1. Document the reason in the PR description

## Debugging

### Script Not Detecting Changes

If the script shows "all files" instead of specific changed files:

- Ensure you're comparing against the correct base branch
- Check that git history is available (CI needs `fetch-depth: 0`)
- Verify `BASE_BRANCH` environment variable is set correctly

### False Positives

If you see coverage regressions that don't make sense:

- Check for rounding differences (0.1% tolerance is built in)
- Verify that coverage was generated correctly
- Compare `coverage/coverage-summary.json` against baseline manually

### Bypassing the Check (Not Recommended)

In exceptional cases, you can update the baseline intentionally. However:

- Document why in the commit message
- Track the increased debt in `quality/gaps.md`
- Get approval from project maintainers
