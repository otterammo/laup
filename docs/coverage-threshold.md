# Coverage Threshold and Diff Coverage Gate (CIG-003 / DOC-620)

## Overview

CIG-003 defines the requirements for enforcing code coverage thresholds in CI and on pull requests.

## Requirements

### 1. Global Coverage Thresholds

Minimum coverage thresholds must be enforced in CI:

- **Lines:** 80%
- **Statements:** 80%
- **Functions:** 80%
- **Branches:** 80%

These thresholds are versioned in `vitest.config.ts` and enforced on every CI run.

### 2. Diff Coverage Gate (PR-Only)

Pull requests must meet additional requirements:

- **Changed Lines Coverage:** 90% minimum
  - Only lines modified in the PR are considered
  - New files must meet the 90% threshold
  - Modified existing files must have 90% coverage on changed lines

### 3. CI Integration

The coverage job in `.github/workflows/ci.yml` must:

- Run `pnpm run test:run --coverage`
- Fail the build if global thresholds are not met
- Upload coverage reports for analysis

### 4. PR Coverage Gate

A separate workflow (`.github/workflows/pr-coverage.yml`) must:

- Run only on pull requests
- Calculate diff coverage for changed lines
- Comment on the PR with coverage results
- Fail if diff coverage is below 90%

## Implementation

### Global Thresholds

Configured in `vitest.config.ts`:

```typescript
coverage: {
  provider: "v8",
  include: ["packages/**/src/**/*.ts"],
  exclude: ["packages/**/src/__tests__/**"],
  thresholds: {
    lines: 80,
    statements: 80,
    functions: 80,
    branches: 80,
  },
}
```

### Diff Coverage Script

The script `scripts/check-diff-coverage.mjs` calculates coverage for changed lines:

1. Uses `git diff` to identify changed lines
1. Parses vitest coverage report (JSON format)
1. Calculates coverage percentage for changed lines only
1. Exits with error code if below threshold

## Usage

### Local Development

Run coverage with threshold checking:

```bash
pnpm run test:run --coverage
```

### CI

Global thresholds are checked in the `quality/coverage` job. Currently set to non-blocking (`continue-on-error: true`) to allow gradual remediation of existing quality gaps. Will be made blocking once all gaps in `quality/gaps.md` are resolved.

### PR Review

The PR coverage gate automatically runs and comments on pull requests with:

- Overall coverage statistics
- Diff coverage percentage
- List of changed files and their coverage
- Pass/fail status

## Rationale

### Why 80% Global Threshold?

The 80% threshold balances:

- **Quality:** Ensures most code paths are tested
- **Pragmatism:** Allows for legitimate uncovered paths (error handling, logging)
- **Consistency:** Aligns with industry standards and QBASE-002 targets

### Why 90% Diff Threshold?

New and modified code should have higher coverage because:

- **Fresh Context:** Developers have the code fresh in mind
- **Active Development:** Easier to write tests during implementation
- **Quality Gate:** Prevents coverage degradation over time
- **Technical Debt:** Existing code may have legitimate reasons for lower coverage

## Exceptions

### Global Threshold Exceptions

If a package cannot meet 80% due to legitimate reasons:

1. Document the exception in `quality/gaps.md`
1. Create a tracking issue
1. Set a target date for remediation

### Diff Threshold Exceptions

Diff coverage can be bypassed with:

1. **PR Label:** `coverage-exception` (requires maintainer approval)
1. **Justification:** Must be documented in PR description
1. **Follow-up Issue:** Must be created for future remediation

Common valid exceptions:

- Generated code
- External API wrappers (mocking difficult)
- Deprecated code being removed soon

## Monitoring

Coverage trends are tracked in:

- `quality/baseline.v1.json` (generated by `pnpm run quality:baseline`)
- `quality/gaps.md` (manual tracking of coverage gaps)
- GitHub PR comments (automatic diff coverage reports)
